    #代码风格

    • 下列位置应该使用空格以使代码结构清晰，提高代码可读性：

    1. 开始、类的定义、以及if、for、do、while、switch、case语句中的代码都要采用缩进方式；

    2. 对齐：程序块的分界符左大括号"{" 和右大括号"}"都另起一行，尾部大括号一行并且位于同一列，同时与引用它们的语句左对齐；对齐只使用TAB键，不使用空格键；不允许把多个短语句写在一行中，即一行只写一条语句；if、for、do、while、case、switch、default等语句自占一行。且都需需要用大括号包裹

    3. 换行：一行的长度超过80个字符需要换行

    4. 二元操作符前后都需要有空格， ‘.’ 除外；

    5. 参数列表中的逗号 ‘,’ 后面；

    6. for 循环中的表达式之间，分号 ‘;’ 后面；

    7. foreach 循环中冒号 ‘:’ 前后；

    8. 方法名 ( 包括声明和调用 ) 和左括号 ‘(’ 之间 不需要 空格；

    9. 小括号与内部字符之间 不需要 空格；

    10. 没有必要 增加若干空格来使某一行的字符与上一行对应位置的字符对齐。

    11. 多行注释中，换行后星号 ‘’ 之前至少放置一个空格，为实现与首行注释的星号 '’ 对齐；

    12. 注释星号 ‘*’ 与注释内容 如果处于同一行，两者 之间放置一个空格；

    13. 注释的双斜线与注释内容之间有且仅有一个空格；

    #编码规范

    #文件命名

    • 资源文件

      1. layout 必须以module名字或缩写作为前缀避免面重复命名

    #类

    • 命名需要使用大驼峰规则命名 注意 关键字 类名 和 {} 之间加一个空格

    • Bean类

      1. 只允许有成员变量和 get set函数

      2. 一般也要加上toString()方法

    • 接口类

      1. 命名必须以大写字母I开头标识这是个接口类

      2. 接口类函数定义不可以使用对方法进行public private等属性修饰

    • 抽象类

      1. 需要以Abs开头或者以Base开头 标识这个是个抽象类或者基类需要子类实现才可以用

    • 测试类

      1. 命名用Test结尾

    • 枚举类

      1. 用enum后缀

      2. 成员全部用大写用下换线分割单词 如 SUCCESS_MSG

    • 异常类

      1. Exception结尾

    • 接口实现类

      1. 接口实现类用 接口名加Impl结尾 如接口为ICacheSeivice 实现类用CacheSeiviceImpl

    • 其他

      1. 类定义用其功能后缀 如Config Dao Service Container Manager Module等

    #变量

    • 命名规范

      1. 常量全部用大写用下换线分割单词 如 SUCCESS_CODE

      2. 成员变量使用小驼峰以m开头如 mTteacherCode

      3. 局部变量使用小驼峰如 teacherCode

      4. bool 可以用 can has is 不必遵循m开头 或者用able结尾 从意义上能区分是bool值就可以

    • 变量排版

      1. Private Protect Public 等同一类权限范围的函数或变量尽量排布在同一个区域内

    • 变量定义

      1. 变量尽量使用有意义的命名拒绝使用魔鬼变量如 list1 list2 data1 data2
      2. 尽量不要使用无法理解的字母缩写如果必须用需要添加必要注释
      3. 定义数组时，方括号 " [] " 是类型的一部分，所以方括号 " [] " 应放在数据类型后，不要放在变量名后
      1. 变量定义一个变量独占一行不允许一行定义多个变量
      1. 修饰符使用规则 如果有多个修饰符，修饰符按照如下顺序排列：

      public protected private abstract static final transient volatile synchronized native strictfp

    #函数

    • 命名及书写规则

      1. 采用小驼峰规则命名public void getData() {}
      2. 注意多个参数逗号后面要加空格 结尾的入参变量和小括号之间不需要空格小括号面的开始大括号和小括号之间加空格函数结束的大括号需要单独一行 如果是空函数可以缩写成{}不需要换行

    • 函数排版

      1. Private Protect Public 等同一类权限范围的函数或变量尽量排布在同一个区域内
      1. 方法声明

      良好的程序设计应该尽可能减小类与类之间耦合，所遵循的经验法则是：尽量限制成员函数的可见性。如果成员函数没必要公有 (public)，就定义为保护 (protected)；没必要保护 (protected)，就定义为私有 (private) 原则就是尽量保证封装性。

      1. 方法排版顺序

      声明顺序：构造方法、静态公共方法、静态私有方法、公共方法、受保护方法、私有方法

      1. 避免参数过多
        一般入参不宜超过五个 如果过多可以考虑将函数拆分 或者封装成对象

      2. 尽量减少使用参数做逻辑控制

      1. 函数定义需遵守单一功能原则

      不要再一个函数里处理过多的逻辑 一个函数只做自己的事就好 如果处理的逻辑过多就要考虑拆分成多个函数

    #文件组织规则

    • 注意文件大小尽量控制文件内代码不要过多，保持文件的单一功能

    • 功能独立切与业务无关的通用功能封装成lib_库

    • 功能独立的业务相关功能代码写成module_库

    • 代码文件结构要划分清楚 根据功能或者模块 划分成不同的目录下 入口类尽量暴露在最外侧方便阅读代码时找到入手点

    #常见用法规则

    • 异常

      1. 捕获异常并不是为了防止程序崩溃 而是要处理它 处理不了的要出log方便定位问题

      2. 可能产生多个异常的代码要分别捕获处理避免使用单一catch来统一处理

      3. 不可用try catch包裹过多的代码尽量精准捕获定位 准确定位问题可能发生的位置

      4. 异常捕获后在catch内添加日志输出并添加方便查验的tag

      5. 集合类变量不可以直接使用isEmpty判断要考虑 null的情况

    • 编程习惯

      1. 多层判断条件不利于代码可读性

      1. map list 数组等 遍历的时候不可以进行添加和删减操作 如果必须做请使用迭代器或者加锁

      2. 类对象尽量减少context上下文的持有，需要时可以在类内部的函数上将context作为入参传入减少持有context造成的内存泄漏 如果必须使用可以考虑持有ApplicationContext.

      3. 使用Handler 尽量使用静态内部类重载Handler对象 并且用弱引用以减少内存泄漏的风险

      4. 处理String 的时候尽量使用 StringBuffer来代替

      5. 函数内定义的局部变量应遵循就近原则 禁止在一个地方定义 在很远的地方才使用 影响代码可读性

      6. 所有重载的方法必须加@Override注解

      7. 不再推荐使用又要兼容旧代码的 变量和方法需要加注解@Deprecated 添加描述注释

      8. 注解需要注意每个注解都要独占一行

      9. 使用while 或者 doWhile 循环的时候必须 结束循环的条件

      10. 严禁使用if (true) while (true)这种实现无限循环

      11. 多线程共享变量时要注意考虑线程安全使用线程安全类型或者加锁

      12. 耗时操作不可以在主线程中进行操作

      13. 代码设计上在耗时操作完成后的回调帮助调用者切回UI线程

      14. 方法入参尽量减少 map Object JsonObject String 来代替Json字符串等无法推断的类型字段名如有特殊需求请添加详细的注释并 提供完整json的定义demo

      15. Android需要传输过程序列化的类推荐使用Parcelable 需要持久化存储的使用Serializable

      16. map遍历使用entrySet代替keySet遍历 减少遍历测试

      17. 包装类如果Integer使用equals比较 基础类型如int可以使用 ==
